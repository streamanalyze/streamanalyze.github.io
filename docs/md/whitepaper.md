href="https://www.streamanalyze.com/" target="_blank"><span style='font-size:
12.0pt;font-family:"Helvetica",sans-serif;mso-fareast-font-family:"Times New Roman";
color:#4183C4;letter-spacing:.15pt'>Stream Analyze Sweden</span></a></span><span
lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>&nbsp;for
easy and flexible development and deployment of applications that process and
analyze real-time streams of data in distributed and mobile environments.
Streams that are produced by sensors on mobile or edge devices can be analyzed
on-line interactively. An edge device can be, e.g., an Android unit, a desktop
computer, a Raspberry Pi, or MCUs like <span class=SpellE>MangOH</span> Red or ARM
M4. <span class=GramE>sa.engine</span> has a very small footprint (from 100kB
to 5MB depending on configuration) and is hardware and OS agnostic and fully
independent of any third party software. <o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>The
combination in <span class=GramE>sa.engine</span> of a main-memory database, a
computational engine, a data stream processor, and an inference engine implemented
on edge devices allows <b style='mso-bidi-font-weight:normal'>edge analytics</b>
directly on the edge devices rather than the contemporary approach of first
uploading all data from devices to a server and then do all processing centrally
on a server or in the cloud. The approach allows for drastic data reduction by
filtering the data streams already in the devices and only population analyses
over collections of devices need to be made centrally. <o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>On
server computers, in order to collect data from edges and combine and forward
the data to other server-based systems and users, <span class=GramE>sa.engine</span>
systems can also be configured as <b style='mso-bidi-font-weight:normal'>stream
servers</b>. For example, whenever the analysis model in some edge device
detects strong vibrations by doing a computation over the latest readings from
its accelerometer, a data stream containing the frequency spectrum of the
vibrations along with the geographical position of the device is transmitted to
a stream server. If the stream server receives many such streams at the same
time from devices in geographical positions close to each other, it may
indicate an earthquake. The stream server may furthermore forward the received
data to other systems, <span class=GramE>e.g.</span> for permanent central
storage or batch analysis.<o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>To
analyze streaming data interactively on a very high and user-oriented level, <span
class=GramE>sa.engine</span> allows analysts and engineers to develop
computations, filters and transformations over data using a&nbsp;<b>query <span
class=SpellE>anguage</span></b>&nbsp;called&nbsp;<b>OSQL </b>(Object Stream
Query Language). With OSQL computations and filters over real-time streaming
data are defined as mathematical formulas and expressions, called&nbsp;<b>stream
models</b>. A stream model is a set of definitions of mathematical functions,
filters, and other expressions over a stream of measurements. With OSQL the
models can be specified on <span class=GramE>a very high</span> level without
deep programming knowledge; you specify&nbsp;<b>what</b>&nbsp;to do rather than
writing detailed programs expressing&nbsp;<b>how</b>&nbsp;to execute the models.
The user needs not worry about details on how to efficiently implement programs
that integrate large numbers of edge devices and servers.<o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>An
OSQL query that continuously makes computations or filtering over measurements
in a data stream is called a&nbsp;<b>continuous query, CQ</b>. <span
class=GramE>sa.engine</span> allows analysts to interactively specify CQs for continuously
analyzing measurements flowing through edge devices and stream servers in
real-time. The result of a CQ is a real-time <b>object stream</b> of processed
and filtered measurements, for example a CQ returning a stream of the position
vectors of a device measured every second when it is close to a given
geo-position.<o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><a name=fusion-queries></a><span lang=EN-US style='font-size:12.0pt;
font-family:"Helvetica",sans-serif;mso-fareast-font-family:"Times New Roman";
color:#333333;letter-spacing:.15pt'>Both stream models and user data are stored
in <span class=GramE>very fast</span> object-oriented <b>in-memory databases</b>
residing on the edge devices and in the stream servers. These databases are important
for data stream processing, which usually involves matching in real-time fast
flowing stream objects against data in a database. For example, to determine
that the frequency spectrum of a measured vibration may later destroy an edge
device, the frequencies measured by vibration sensors on the device is matched
against a local database of resonance frequencies in the device.<o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>The
system includes a library of over 1000 predefined OSQL-functions for math/stat
computations, object stream filtering and transformation, signal processing, <span
class=GramE>model</span> and data management, and much more. The function
library is continuously extended for new customer needs and it is easy to
define and deploy new user functions on-the-fly.<o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>In
order to combine object streams from several edges, <span class=GramE>sa.engine</span>
supports&nbsp;<b>fusion queries</b>&nbsp;that combine object streams. An
example of a fusion query is a CQ observing when several edge devices in an
area detect strong vibrations at the same time. The user is alerted when the
fusion query produces results, perhaps together with a visualization in
real-time of the magnitude of the observed vibrations. A user can then
interactively send new queries on-the-fly to <span class=GramE>affected</span>
devices to find out details of their causes. <o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>Fusion
queries require the integrated data streams to be comparable even though the involved
object streams may represent the same or similar data in different ways. For
example, one device may represent temperature in Fahrenheit while another one
use Celsius. To be able to combine such <b>heterogeneous</b> data streams from
different devices, <span class=GramE>sa.engine</span> allows <b>mediator </b>models
to be defined as queries and functions that harmonize arriving heterogeneous object
streams by transforming them to a universal model, called an <b>ontology</b>,
in stream servers that integrate data streams from different edges. Mediation
can be, e.g., mapping local names of sensors to a universally known nomenclature,
measurement unit conversions, and calibrations of local measurements.<o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>Existing
algorithms and libraries implemented in a regular programming language, such as
C or Java, can be plugged into the system as <i>foreign OSQL functions</i>
using programming language specific APIs. The foreign functions can then be
transparently used in OSQL queries and expressions.<span
style='mso-spacerun:yes'> </span>For example, a large library of basic
mathematical, statistical, and machine learning algorithms are implemented as
foreign OSQL functions in C.<o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>In
order to access external data streams, <span class=GramE>sa.engine</span>
provides <b>wrapper</b> functionality, which is APIs that enable mapping over incoming
data stream objects as they arrive in order to inject them into the sa.engine
kernel so that the accessed stream can be used in CQs expressed in OSQL. Different
streams often represent their elements using different physical data
structures, so the wrapper functionality includes the ability to extend <span
class=GramE>sa.engine</span> with new physical data representations and to
convert the external data representations to a format already supported by the
system. The wrappers are defined as OSQL functions that return object streams
from wrapped data sources. There is a library of predefined wrappers to
interoperate with common data infrastructures such as relational databases
through&nbsp;JDBC and data processing systems through Kafka, Azure IoT Hub, or
MQTT. New wrappers can easily be developed.<o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>Common
machine learning algorithms such as </span><span lang=EN-US style='color:black;
mso-color-alt:windowtext'><a href="https://en.wikipedia.org/wiki/DBSCAN"><span
style='font-size:12.0pt;font-family:"Helvetica",sans-serif;mso-fareast-font-family:
"Times New Roman";letter-spacing:.15pt'>DBSCAN</span></a></span><span
lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>,
</span><span lang=EN-US style='color:black;mso-color-alt:windowtext'><a
href="https://archive.siam.org/meetings/sdm06/proceedings/030caof.pdf"><span
class=SpellE><span style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";letter-spacing:.15pt'>DenStream</span></span></a></span><span
lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>,
</span><span lang=EN-US style='color:black;mso-color-alt:windowtext'><a
href="https://en.wikipedia.org/wiki/K-nearest_neighbors_algorithm"><span
style='font-size:12.0pt;font-family:"Helvetica",sans-serif;mso-fareast-font-family:
"Times New Roman";letter-spacing:.15pt'>k-NN</span></a></span><span lang=EN-US
style='font-size:12.0pt;font-family:"Helvetica",sans-serif;mso-fareast-font-family:
"Times New Roman";color:#333333;letter-spacing:.15pt'>, and </span><span
lang=EN-US style='color:black;mso-color-alt:windowtext'><a
href="https://en.wikipedia.org/wiki/K-means_clustering"><span style='font-size:
12.0pt;font-family:"Helvetica",sans-serif;mso-fareast-font-family:"Times New Roman";
letter-spacing:.15pt'>k-means</span></a></span><span lang=EN-US
style='font-size:12.0pt;font-family:"Helvetica",sans-serif;mso-fareast-font-family:
"Times New Roman";color:#333333;letter-spacing:.15pt'> are available as predefined
OSQL models and the user can easily extend this with other algorithms defined in
OSQL or as foreign functions. Learning and inference are supported on both edges
and servers.<o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>Machine
learning requires pre-processing of sensor data before applying the learned
inference algorithm, followed by post-processing of the inferred knowledge.
With <span class=GramE>sa.engine</span> both pre and post-processing are easily
expressed using the powerful object stream filtering, transformation and
math/stat analytics capabilities of OSQL.<o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>The
system includes a deep learning subsystem, SANN, where neural network models
defined by <span class=SpellE>Tensorflow</span>/<span class=SpellE>Tensorboard</span>
can be automatically translated into an internal binary SANN representation and
pushed out to the edge databases. Once stored in an edge database the SANN
inference engine can analyze local sensor readings to detect anomalies. <o:p></o:p></span></p>

<p class=MsoNormal style='margin-bottom:10.2pt;line-height:normal;background:
white'><span lang=EN-US style='font-size:12.0pt;font-family:"Helvetica",sans-serif;
mso-fareast-font-family:"Times New Roman";color:#333333;letter-spacing:.15pt'>SANN
furthermore allows to continue training and even building neural networks on
edge devices. Thus, centrally trained models can be further retrained and
modified on edge devices to adapt their behavior to their environment.</span></p>
